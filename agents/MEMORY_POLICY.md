# Memory Policy

## Purpose
Keep memory useful, scoped, low-noise, and scalable.

## Memory Hierarchy
1. Global governance
2. Company
3. Project
4. Restricted shared
5. Agent private
6. Agent experience
7. User
8. Session
9. Ephemeral
10. Cold storage

## Store
- verified findings
- blockers
- important decisions
- reusable lessons
- handoff-critical context
- durable company methods
- project bottlenecks worth preserving

## Do Not Store
- raw chatter
- duplicate updates
- speculative noise
- low-value logs
- stale project residue
- every intermediate thought

## Access Rules
- Only operatives and orchestrator may access global governance memory.
- Only orchestrator creates new shared, company, project, restricted, or archival memory spaces.
- Agent private memory is respected by default; orchestrator may inspect it only when needed.
- Completed project memory moves to cold storage and is isolated from ordinary agents.

## Write Rules
### Direct writes
- own private memory
- assigned project memory when useful and non-noisy

### Proposed writes
- company memory
- restricted shared memory

### Orchestrator-only writes
- global governance memory
- creation of new memory spaces
- archival metadata

## Promotion Rules
- private -> project when useful to collaborators
- project -> company when reusable across projects
- company -> governance when it becomes a system-wide rule

## Project Completion
- archive project memory to cold storage
- extract bottlenecks/issues into relevant specialist experience.md
- wipe most specialized agent project memory after completion
- carry forward only curated experience.md knowledge to the next project
- revoke routine agent access to archived project memory
